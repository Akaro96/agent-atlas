# Positioning

## Core Pitch

**Agent Atlas** is a portable dual-agent workspace harness for people who want a serious daily driver, not a giant performance piece.

## Why Someone Would Choose This Instead Of A Bigger Competitor

### 1. Clearer real-world boundaries

This kit separates:

- operational workspace
- durable knowledge
- live external docs

Many harnesses blur those together.

### 2. Better fit for users who mix tools

This project explicitly supports:

- Codex
- Claude Code
- optional Obsidian

Many public harnesses are effectively single-agent systems with add-ons.

### 3. Less overhead

If you do not want hundreds of commands, dozens of agents, and a framework you have to maintain full-time, this is the stronger choice.

### 4. More auditable

This repository includes:

- install flow
- validation
- smoke tests
- portable helper tools
- a lightweight knowledge compiler flow

It is meant to be checked, not merely admired.

### 5. Better Windows story

A lot of public harness repos feel Linux-first even when they claim portability.
This kit is explicit about Windows instead of pretending shell differences do not matter.

## Where Competitors Are Still Stronger

### `clarc`

Stronger at:

- raw breadth
- command surface
- workflow operating system feel

This project is stronger at:

- simplicity
- lower maintenance overhead
- dual-agent balance

### `everything-claude-code`

Stronger at:

- ecosystem breadth
- sheer quantity of modules
- public “big framework” impression

This project is stronger at:

- tighter packaging
- less sprawl
- clearer portable distribution boundaries

### `cco`

Stronger at:

- containment and security-first positioning

This project is stronger at:

- day-to-day usability
- general workspace orchestration

## Best Target User

The ideal adopter is someone who thinks:

> I want my AI workspace to be clean, portable, understandable, and powerful enough to use every day, but I do not want to maintain a giant meta-framework.

## Why This Can Beat More Famous Systems For Some Users

Because it combines:

- less framework weight
- stronger daily usability
- clearer project/vault boundaries
- real validation
- an explicit knowledge pipeline

For users who value those things more than raw feature count, this is the stronger choice.
